# Анализатор Шрифт РЖЯ

Интерактивный учебный веб-анализатор записи жестов в системе **Шрифт РЖЯ**. Пользователь вводит строку в обычных символах клавиатуры, а приложение показывает:

- какие токены и группы токенов распознаны;
- как запись раскладывается на кадры, таймлайны и немануальные блоки;
- какие найдены конфигурации рук, ориентации, локализации, контакты, движения и служебные символы;
- где есть неизвестные символы, неполные группы или подозрительные последовательности;
- в простых случаях — смысловое описание жеста на русском языке;
- JSON-разбор, пригодный для дальнейшей обработки.

Проект не является переводчиком РЖЯ на русский язык. Главная задача проекта — структурный анализ записи.

## Лингвистический контекст

Шрифт РЖЯ основан на ASL Font и адаптирован под русский жестовый язык и русскую раскладку ЙЦУКЕН. В этой системе символы клавиатуры служат кодами графических знаков: конфигурации рук чаще всего задаются одиночными символами, а ориентации, локализации, движения и немануальные компоненты часто записываются цепочками. Поэтому анализатор не может быть простой таблицей «символ → значение»: ему нужен парсер, который ищет многосимвольные паттерны и собирает их в иерархическую структуру.

Базовая модель разбора опирается на идею чередования **кадров** и **таймлайнов**: кадр описывает положение рук, ориентации и локализации, а таймлайн описывает движение между кадрами. Немануальные компоненты выделяются отдельно, потому что они описывают выражение лица, движения глаз, головы, плеч и другие неручные параметры.

Значения символов (конфигурации, ориентации, локализации, контакты, движения, модификаторы, движение неведущей руки, немануальные компоненты) взяты из справочных материалов Шрифта РЖЯ: страница `reference.html`, уроки  и справочные таблицы в репозитории AjnoEO/rslfont. 

## Кому полезен проект

- студентам, которые изучают систему записи Шрифта РЖЯ;
- авторам учебных материалов и демонстраций;
- исследователям РЖЯ, которым нужно быстро проверить структуру записи;
- разработчикам словарей и инструментов, где нужна предварительная токенизация записи жеста;
- пользователям, которые набирают записи вручную и хотят видеть ошибки ввода.

## Возможности

- ввод записи Шрифта РЖЯ в текстовое поле;
- подсветка токенов разными CSS-классами по типам компонентов;
- сохранение исходной строки при подсветке, включая пробелы и переносы;
- интерактивный выбор токена: по клику показывается объяснение выбранного элемента;
- иерархический разбор: немануальные блоки, кадры, таймлайны, руки, локализации, контакты, движения;
- бета-блок смыслового описания для простых паттернов: сначала строится промежуточный semantic layer, затем короткое русское объяснение с понятными формулировками контакта, локализации, ориентации руки и направления движения;
- валидатор с уровнями `ok`, `info`, `warning`, `error`;
- обработка пустого ввода, неизвестных символов, незакрытых скобок, одиночных частей ориентации, нераспознанных групп и невидимых разделителей из скопированного текста;
- распознавание неоднозначных символов (`!` и `;`) с объяснением выбранной интерпретации;
- блок «Возможные последовательности» для технически неоднозначных разборов;
- компактный JSON-разбор обычным шрифтом;
- набор автотестов для парсера и валидатора.

## Запуск тестов

```bash
npm install
npm start
npm test
```

Тесты покрывают обязательные учебные сценарии:

- пустая строка;
- корректная запись;
- запись с неизвестным символом;
- запись с несколькими компонентами;
- запись с нарушенной структурой;
- длинная строка;
- кириллица и служебные ASCII-символы, которые являются кодами Шрифта РЖЯ;
- невидимые соединители Unicode из скопированных примеров;
- дополнительные локализации `9ЁЙЁ0` и локализации на кисти из справки, например `9Ёв0`, `9.в0`;
- сжатые немануальные цепочки вроде `:%ЪХ%ХЪ`;
- бета-семантика для простого паттерна и отказ для сложного;
- немануальные движения и уточнённое круговое движение;
- неоднозначные последовательности.

## Примеры ввода

```text
(ЁЁ)*.Эо
```

Пример корректной записи: локализация, контакт, ориентация и конфигурация руки.

```text
."Ч_Ё=7_Ё=Ч
```

Пример с несколькими кадрами и таймлайнами.

```text
6
```

Пример одиночной конфигурации руки. 

```text
9abc0
```

Пример неизвестной группы `9...0`: анализатор сохраняет её одним токеном и показывает предупреждение.

```text
(ЁЁ
```

Пример нарушенной структуры: открывающая скобка есть, закрывающей нет.

```text
Q#
```

Пример с неизвестными символами: `Q` и `#` — чужие символы.

```text
БЖБ9Б0(Д_)
```

Пример немануальных движений и уточнённого кругового движения.

```text
9.в0;5Д
```

Пример локализации на кисти неведущей руки из справки: тыльная сторона ладони.

```text
а9ЁЙЁ0а/
```

Пример с локализацией «обе стороны груди» (`9ЁЙЁ0`).

```text
9ЁЙ0**=Ёф
```

Пример из урока 9 с локализацией левой стороны груди и повторным контактом.

```text
:%ЪХ%ХЪ
```

Пример сжатой немануальной артикуляции.

## Как работает парсер

```text
строка пользователя
  → нормализация Unicode NFC
  → удаление пробелов и невидимых Unicode-разделителей для машинного разбора
  → токенизация longest-match-first
  → распознавание строго ограниченных динамических паттернов
  → группировка в кадры, таймлайны и немануальные блоки
  → бета semantic layer для простых паттернов
  → выделение рук внутри кадра
  → валидация
  → подсветка + дерево + JSON
```

Стратегия `longest-match-first` нужна, чтобы длинные цепочки вроде `9Ё10`, `9;0`, `:%(*)` или `(Д_)` не распадались на отдельные символы. После фиксированных совпадений парсер запускает распознаватели для относительных локализаций, локусов и сжатых немануальных цепочек.

## JSON-формат результата

`parseSign()` возвращает объект со следующими ключевыми полями:

- `rawInput` — исходная строка пользователя;
- `normalized` — строка без пробелов, используемая для анализа;
- `tokens` — список лексических токенов;
- `units` — иерархические единицы разбора;
- `frames` — кадры;
- `timelines` — таймлайны;
- `nonmanualUnits` — немануальные блоки;
- `components` — найденные компоненты по типам;
- `unknown` — неизвестные символы и нераспознанные группы;
- `warnings` и `errors` — сообщения валидатора;
- `possibleSequences` — технически возможные варианты последовательности при неоднозначных символах;
- `summary` — краткая статистика.

Внутренний объект `parseSign()` подробный, потому что он нужен UI и тестам. В панели интерфейса и при копировании выводится компактная версия: без повторяющихся вложенных ссылок на одни и те же токены, но с токенами, единицами, диагностикой, вариантами последовательности и semantic layer. JSON выводится обычным моноширинным шрифтом. 

## Бета semantic layer

Файл `src/semantic.js` добавляет промежуточный слой между фонетическим разбором и русским объяснением. Он собирает кадр в поля `location`, `dominantHand`, `nonDominantHand`, `contact`, `timeline`, а затем строит короткое объяснение.

Бета-модуль поддерживает простые случаи: один кадр, не больше одного таймлайна, до двух рук, одна локализация, простое направление или круговое движение. В поддержанных случаях он различает положение около части тела и контакт с ней: например, `9Й0` без контакта описывается как положение **у груди**, а `9Й0*...` — как касание **груди**. Направления даются с точки зрения говорящего: `/` описывается как движение **от себя**, а `З` — как движение **к себе**. Ориентации, локализации и немануальные сочетания сопровождаются расшифровкой. Если запись содержит несколько кадров, неизвестные группы, неполную ориентацию, сложную траекторию или другой неподдержанный элемент, блок показывает причины отказа.

Для повторений в немануальных компонентах используется алгоритм на основе базовых слогов из справки (`МИ`, `ПАФ`, `СТА`, `ЧУ`, `НА`, `ВА`). Поэтому сокращения вроде `:%ЪЭХ%%(ЪЭ` читаются как `НА-НА`, а `:%ЪЭХ%%%(ЪЭ` — как `НА-НА-НА`; новые слоги при этом не добавляются.

В многокадровых записях парсер учитывает сокращение: если в следующем кадре дана только новая локализация или новая ориентация, в `effectiveHands` сохраняются руки из предыдущего кадра. Валидатор всё равно оставляет предупреждение, чтобы пользователь мог проверить, не потерялась ли конфигурация при вводе.

## Подключение шрифта

Шрифт РЖЯ включён прямо в репозиторий — в папке `assets/fonts/` (`rsl-font.ttf` вместе с `license.txt` и `readme.txt`). В `src/styles.css` он подключается через `@font-face` с тремя источниками, которые браузер пробует по порядку:

```css
@font-face {
  font-family: "RSLFont";
  src: local("Ajno"), local("RSL Font"),
       url("../assets/fonts/rsl-font.ttf") format("truetype"),
       url("https://cdn.jsdelivr.net/gh/AjnoEO/rslfont@master/RSL_font/symbol_font_for_rsl/rsl-font.ttf") format("truetype");
  font-display: swap;
}
```

1. уже установленный в системе шрифт; 2. копия из репозитория (работает офлайн); 3. запасной вариант с CDN jsDelivr напрямую из репозитория AjnoEO/rslfont. Если ни один источник недоступен, текст показывается системным символьным шрифтом — запись остаётся читаемой как обычные символы.

Шрифт распространяется по лицензии Creative Commons Attribution 3.0 и принадлежит проекту AjnoEO/rslfont (клон «Symbol Font For ASL»). Лицензия разрешает включать файл в репозиторий при условии сохранения атрибуции и сопроводительных файлов — подробности в `NOTICE.md` и в `assets/fonts/`.

## Ограничения

- Анализатор не переводит жесты на русский язык и не определяет лемму жеста.
- Справочники покрывают базовые и частые паттерны, но не являются полной нормативной грамматикой Шрифта РЖЯ.
- Для многих конфигураций указано обобщённое текстовое описание: точная форма должна читаться по самому знаку шрифта.
- Некоторые редкие или авторские цепочки сохраняются как `нераспознанная группа`, чтобы не ломать разбор.
- Semantic layer находится в бета-режиме и описывает только простые записи. Для сложных записей он явно сообщает, почему описание не поддержано.
- Возможные последовательности пока перечисляются только для технически неоднозначных `!` и `;`, а не для всех лингвистических вариантов записи.
- Поиск по сходству жестов и словарная выдача не реализованы: это отдельная задача уровня двустороннего словаря.
